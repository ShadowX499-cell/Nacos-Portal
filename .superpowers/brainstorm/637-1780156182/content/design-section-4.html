Design — Section 4 of 4: Edge Cases, Error Handling & Testing

The tricky parts that break if not thought through upfront.

Critical edge cases

⚠️ Random ID collision

crypto.randomInt(10000, 99999) gives 90,000 slots per program per year. At 500 students/year per program, collision probability is <0.3%. Service retries up to 5× with fresh random. After 5 failures → error ID_GENERATION_FAILED. Practically impossible to hit but safely handled.

⚠️ Session advance with unpublished gradebooks

If there are draft/in-progress gradebooks for the current semester, the preview endpoint warns: "3 gradebooks still in draft — advancing now means those results won't be captured in carryover computation." HOD must acknowledge this warning in the confirm payload (force: true) to proceed.

⚠️ Carryover student passes the course in a later session

When a gradebook is published and a student's new grade ≥ 40 (pass) for a course that had isCarryover = true, the service stamps resolvedAt on the old StudentGrade row. If all carry-overs are resolved, studentStatus reverts to active. The transcript shows both the original fail and the later pass.

⚠️ Photo upload — file validation

Multer config: JPEG/PNG only, max 2 MB, single file. On validation failure → 400 INVALID_FILE_TYPE or FILE_TOO_LARGE. File is saved with a UUID filename (not the original name) to prevent path traversal. Old photo file is deleted when replaced.

⚠️ Permission escalation attempt

A course_adviser or result_exam_officer hitting an HOD-only endpoint returns 403 FORBIDDEN_ROLE. The requirePermission() middleware checks both role and superAdminType and logs the failed attempt to audit_logs with action UNAUTHORIZED_ACCESS_ATTEMPT.

⚠️ Double-advance guard

The advance endpoint checks if an AcademicTransition already exists for the target session/semester combo. If yes → 409 ALREADY_ADVANCED. Prevents accidental double-click from promoting students twice.

Testing strategy

Backend (Jest)
  • requirePermission middleware — all 3 roles × all protected actions
  • ID generation — collision retry logic
  • advanceSemester service — level promotion, graduation, carryover computation
  • publishGradebook side-effects — carryover flagging + resolution
  • Revenue aggregation — filter combinations
  • Audit log — written on every mutating operation
Frontend (Vitest + RTL)
  • CreateUserPage — photo upload preview, state→LGA cascade
  • UserListPage — program tab filter, carryover badge rendering
  • Dashboard payment badges — 0/8, 2/8, 8/8 states
  • AcademicCalendarPage — preview modal, confirmation flow
  • Role-gated sidebar items — hidden for non-HOD

Audit log coverage — every super admin action is logged

STUDENT_CREATED       · actor, studentId, program, level
STUDENT_VALIDATED     · actor, studentId, old status → new status
STUDENT_SUSPENDED     · actor, studentId, reason
GRADE_ENTERED         · actor, courseId, studentId, old score → new score
RESULT_PUBLISHED      · actor, gradebookId, carryovers_flagged count
SESSION_ADVANCED      · actor, from → to, students_promoted, graduates
SUPER_ADMIN_CREATED   · actor (HOD), target, role assigned
SUPER_ADMIN_REVOKED   · actor (HOD), target, role removed
ELECTION_CREATED      · actor, electionId, title
PAYMENT_VERIFIED      · actor, paymentId, studentId, type, amount
UNAUTHORIZED_ATTEMPT  · actor, attempted action, their role
